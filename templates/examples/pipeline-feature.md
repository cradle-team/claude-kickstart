# 機能開発パイプライン例

Worker/Manager品質ゲートパターンを使った機能開発の具体例。

## 概要

新機能の設計→実装→セキュリティレビューを段階的に実行する。
各ステージで品質ゲートを通過しないと次に進めない。

## ステージ構成

### Stage 1: 設計
- **Worker**: Architect Agent
  - 要件分析、技術選択、コンポーネント設計、API設計
  - 出力: 技術設計書（概要、アーキテクチャ、データモデル、タスク分割）
- **Manager**: Design Reviewer
  - 採点基準（各10点満点）:
    1. 要件充足: 全要件をカバーしているか
    2. 拡張性: 将来の変更に耐えうるか（ただし過度な抽象化はNG）
    3. シンプルさ: 必要最小限の複雑さか（YAGNI）
  - 閾値: 全項目8/10以上で合格
  - 差し戻し時: 「要件Xが未対応」「設計が過度に複雑」等を具体指摘

### Stage 2: 実装
- **Worker**: Coder Agent
  - 設計書に基づいてコードを書く
  - テストも書く（TDD推奨）
  - PRを作成
- **Manager**: Code Reviewer
  - 採点基準（各10点満点）:
    1. 型安全性: `any`不使用、適切な型定義
    2. テストカバレッジ: 正常系・異常系・エッジケース
    3. 可読性: 命名、構造、コメント
    4. 規約遵守: プロジェクトのルールに準拠
  - 閾値: 全項目8/10以上で合格

### Stage 3: セキュリティレビュー
- **Worker/Manager兼任**: Security Reviewer Agent
  - OWASP Top 10チェック
  - シークレット漏洩チェック
  - 認証/認可の検証
  - 出力: セキュリティレビュー結果（PASS/WARNING/BLOCK）
  - BLOCKの場合はStage 2に差し戻し

### Stage 4: 人間が最終確認・マージ

## 実行手順

1. リーダーがAgent toolでArchitect Agentを起動
2. 設計書をDesign Reviewerに渡して採点
3. 合格後、Coder Agentを起動（設計書を渡す）
4. PRをCode Reviewerに渡して採点
5. 合格後、Security Reviewerを起動
6. PASSなら人間に最終確認を依頼

## 注意事項

- Stage 2でセキュリティ問題が見つかったら、Stage 2のCoderに差し戻し
- 設計の根本的な問題が見つかったらStage 1からやり直し
- 各ステージの成果物はgitでコミットして記録を残す
